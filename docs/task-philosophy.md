# task-philosophy.md

## Propósito

Este sistema existe para ayudar a capturar, aclarar, estimar, agendar y revisar trabajo personal de forma realista. No busca solo registrar pendientes, sino apoyar mejores decisiones sobre qué hacer, cuándo hacerlo y cuánto podría tomar.

El objetivo no es producir planes perfectos. El objetivo es reducir fricción, hacer visible el trabajo real y aprender con el tiempo a partir de lo que de verdad ocurre.

## Principios

### 1. Capturar primero, aclarar después
La captura debe ser rápida y de baja fricción. Una entrada puede llegar incompleta, vaga o mal formulada. El sistema debe aceptarla sin exigir estructura perfecta desde el inicio.

### 2. No todo lo capturado está listo para estimarse o agendarse
Capturar algo no significa que ya sea una tarea clara. Antes de estimar con confianza o reservar un bloque en agenda, el sistema debe evaluar si la entrada representa una acción concreta, un proyecto, una idea, un recordatorio o algo que aún necesita aclaración.

### 3. Estimar acciones claras, no intenciones vagas
Las estimaciones son más útiles cuando se aplican a acciones observables y ejecutables. Si una entrada es ambigua, la respuesta correcta no es fingir precisión, sino bajar la confianza, pedir aclaración o sugerir división.

### 4. Planear por capacidad real, no por optimismo
La agenda debe reflejar límites reales de tiempo, energía, atención, contexto y dependencias. Que algo importe no significa que quepa hoy. Una buena planificación protege la capacidad, no solo expresa deseos.

### 5. Usar rangos y niveles de confianza
El sistema debe evitar la falsa precisión. Siempre que sea posible, debe pensar en rangos, supuestos y niveles de confianza, especialmente en tareas ambiguas, creativas o poco frecuentes.

### 6. La posposición es información
Cuando una tarea se mueve varias veces, eso no debe interpretarse automáticamente como falta de disciplina. Puede indicar mala definición, fricción emocional, prioridad inflada, dependencia externa, mal momento o mala estimación.

### 7. El sistema debe registrar más de lo que pregunta
Para no volverse pesado, el sistema debe pedir al usuario la menor cantidad de información posible e inferir el resto cuando sea razonable. La riqueza del sistema debe vivir más en su interpretación y registro interno que en formularios largos.

### 8. El historial importa tanto como el estado actual
No basta con saber qué está pendiente. También importa cómo cambió una tarea, cuántas veces se movió, si fue redefinida, cuánto tomó realmente y por qué se desvió de lo esperado.

### 9. Aprender de la realidad sin volverse dogmático
Las estimaciones futuras deben mejorar a partir de diferencias entre lo estimado y lo observado. Ese aprendizaje debe orientar, no imponerse rígidamente. El sistema debe sugerir con criterio, no volverse un obstáculo inflexible.

### 10. Revisar también es trabajar
Las tareas mal definidas, estancadas, repetidamente pospuestas o sistemáticamente subestimadas requieren revisión. Revisar no es un castigo, sino parte normal de mantener un sistema confiable.

## Qué tomamos de otros sistemas

### De Getting Things Done
- Captura rápida y confiable
- Diferencia entre proyectos y siguientes acciones
- Importancia de aclarar qué significa realmente cada pendiente
- Valor de contextos, dependencias y estados como “esperando respuesta”

### De Bullet Journal
- Captura simple y flexible
- Migración como señal de valor, fricción o resistencia
- Distinción entre tareas, eventos y notas
- Revisión periódica como herramienta de conciencia

### De Time Blocking
- El trabajo necesita contenedores reales en el tiempo
- La capacidad diaria es limitada
- No todos los bloques sirven para el mismo tipo de tarea
- Los buffers y márgenes son parte del trabajo real, no desperdicio

## Cómo debe pensar el sistema

Cuando entra una nueva entrada, el sistema debe intentar responder:

- ¿esto es una tarea, un proyecto, una idea, una nota o un recordatorio?
- ¿la siguiente acción está clara?
- ¿esto cabe en un bloque razonable?
- ¿requiere energía baja, media o alta?
- ¿depende de alguien más?
- ¿tiene un mejor contexto o momento?
- ¿conviene hacer, agendar, dividir, incubar o esperar?

## Qué significa una buena estimación

Una buena estimación no es la que acierta un número exacto. Es la que mejora una decisión.

Una estimación es útil si ayuda a:
- saber si algo cabe hoy
- escoger un bloque razonable
- detectar riesgo de expansión
- notar que algo debe dividirse
- comparar opciones de agenda con más realismo

## Qué debe aprender el sistema

Con el tiempo, el sistema debe detectar patrones como:
- tipos de tarea que suelen durar más o menos de lo previsto
- tareas que se expanden con frecuencia
- contextos donde cierto trabajo sale mejor
- franjas horarias más favorables según tipo de trabajo
- señales tempranas de ambigüedad o fricción
- diferencias entre estimación inicial, revisión posterior y duración real

## Regla de humildad

El sistema debe ayudar a pensar mejor, no reemplazar el juicio del usuario. Puede sugerir, advertir y aprender, pero no debe imponer rigidez innecesaria ni hacer más difícil capturar y actuar.

## Evolución

Este documento está vivo. Debe actualizarse cuando aparezcan nuevos patrones, mejores heurísticas o cambios en la forma real de trabajar del usuario.
